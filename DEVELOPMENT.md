# Development notes

This file is for contributors and maintainers -- implementation rationale, verification history, and known limitations that don't belong in the user-facing README. If you just want to *use* this library, see [README.md](README.md) instead.

---

## Design rationale: long-run / unattended operation

`begin()` and `loop()` self-heal from network trouble in two tiers, deliberately kept separate because they have very different costs:

1. **Wi-Fi down**: after ~1 min (`setWifiHardReinitAfter()`), a full Wi-Fi radio reinit -- clears a wedged module/driver state a plain retry can't. After ~5 min (`setWifiForceResetAfter()`) still down, a full device reset. Neither ever contacts Azure -- DPS isn't reached until Wi-Fi is actually up -- so both thresholds can be as tight as needed with no downside.
2. **DPS provisioning failing** (Wi-Fi fine, Azure/config the problem): exponential backoff (`setProvisionRetryInitial()`/`setProvisionRetryMax()`, default 1min doubling to 15min) instead of resetting. A reset can't fix a wrong ID scope or missing TLS cert any faster than just retrying, and resetting on a tight timer here risks Azure rate-limiting the device during a real outage. Worst case (permanent misconfiguration): a slow, harmless retry loop forever, not a bricked or spamming device.

The reset itself uses `RSTCTRL.SWRR` on megaAVR (Uno WiFi Rev2's dedicated software-reset register, chosen specifically to avoid the watchdog's documented timing-accuracy quirks on that chip -- see `reset.cpp`) and `ESP.restart()` on ESP32, which has no equivalent timing concern.

## Design rationale: hardware watchdog (`enableWatchdog()`)

Petting only happens in two places, and *why* matters:

- **Inside `connectWiFi()`'s Wi-Fi-reconnect wait loop.** This loop can legitimately block for minutes (the reinit/reset tiers above) -- without petting here, a real Wi-Fi outage would trigger a watchdog reset within ~8s, cutting off that carefully-tuned self-healing sequence entirely. A design flaw was caught and fixed before shipping: the first instinct was to pet only once per `AzureIoT.loop()` call, which would have caused exactly this conflict.
- **Deliberately NOT inside the "waiting for network time" loop.** That loop has no escalation tiers of its own -- if NTP is genuinely unreachable (Wi-Fi connected, no route to the internet), it would otherwise hang forever with zero protection. Leaving it un-petted means the watchdog (if enabled) is the one thing that can still catch and recover from that specific case.

~8 seconds is megaAVR-0's own maximum reliable watchdog period (`WDT_PERIOD_8KCLK_gc`), used on both platforms for consistency rather than picking a different, harder-to-reason-about number for ESP32.

## Design rationale: remote telemetry polling (`onRemoteTelemetry()`, v1.1)

This feature exists because devices cannot subscribe to another device's telemetry directly over MQTT -- Azure deliberately scopes each device's credentials to only its own data, as a security boundary, not a limitation of this library. The only way to bridge two devices is to poll Azure's REST API from the cloud side (or relay through a Rule/workflow, which was considered and rejected for this library specifically -- see below).

**Why polling, not the Rules/Power-Automate relay that was also discussed:** the relay approach needs zero new device-side code (it reuses `onBoolProperty()`, already built) and reacts instantly rather than on a delay -- objectively better on both those axes. It was set aside because it requires setting up a separate Azure workflow service (Power Automate or Logic Apps), which is real, one-time configuration work outside this library's scope entirely. Direct polling was chosen so the *entire* feature -- setup and all -- stays inside "things this library and its examples can do," at the cost of a real delay and a new credential.

**Why a hard 1000ms floor, enforced in code rather than just documented:** every other tunable in this library (send interval, Wi-Fi thresholds, DPS backoff) only affects *this device's* own resource usage if set too aggressively. This one is different -- IoT Central's REST API has a documented, hard limit of **20 requests/second per application**, shared across every caller (every device polling, plus dashboards, plus any other integration touching the same app). A device set to poll too fast doesn't just strain itself, it can push the *entire application* into 429 (rate-limited) territory for everyone. That asymmetry -- one misconfigured device affecting unrelated users of the same app -- is why this is the one setter in the library that clamps rather than trusts the caller.

**Why a separate credential (API token) instead of reusing the device's own Connect credentials:** confirmed directly against Microsoft's own REST API authentication docs -- the API only ever accepts a bearer token or an IoT Central API token, never a device's own SAS-token-based identity. This isn't a design choice this library made; it's the only mechanism Azure exposes for this.

**Why "App Operator" is recommended, not "App Administrator":** confirmed via the actual REST API response for `GET /api/roles` -- Administrator and Builder both carry template-editing/app-administration rights this feature never needs; Operator is the least-privileged built-in role that can still read telemetry.

## Known limitations

- **Peak stack depth on Uno WiFi Rev2** during DPS provisioning is ~2.2-2.6KB on a board with 6KB total SRAM (measured with `avr-gcc -fstack-usage` against the real functions, not estimated) -- but this only happens once, at boot, before the MQTT heap buffer even exists yet. Not a concern on ESP32 (hundreds of KB of RAM). Not endurance-tested on real Uno WiFi Rev2 hardware yet.
- **`reset.cpp`'s Uno WiFi Rev2 branch is not compile-tested** in the environment this was built in -- it needs the real `atmega4809` device header, only available through Arduino's own Board Manager toolchain. Confirmed against two independent sources (the official datasheet and Arduino's own bootloader source for this chip) but not built or run for real. The ESP32 branch (`ESP.restart()`) was compiled and run against the real Arduino-ESP32 core's documented API, but likewise not on physical ESP32 hardware. Test both once on real hardware -- deliberately hang the sketch and confirm it actually resets -- before depending on either for unattended operation.
- **`platformWatchdogEnable()`/`platformWatchdogPet()` on megaAVR-0 -- CONFIRMED WORKING on real Uno WiFi Rev2 hardware.** A real user test: normal operation ran cleanly with the watchdog enabled (no false-positive resets), then a deliberate infinite-loop hang was triggered on command, and the board reset itself and reconnected automatically within the expected window -- no button press, no re-upload, entirely on its own. This confirms `_PROTECTED_WRITE(WDT.CTRLA, WDT_PERIOD_8KCLK_gc)` works correctly on real megaAVR-0 hardware, not just against documentation. **Still not verified on real hardware: ESP32's watchdog path** (`esp_task_wdt_config_t` -- no ESP32 hardware test reported yet) and `reset.cpp`'s `RSTCTRL.SWRR` path specifically when triggered by the *other* self-healing tiers (Wi-Fi down for 5+ minutes) rather than the watchdog -- the watchdog test above exercises the same underlying reset mechanism, but the Wi-Fi-down-specific trigger path hasn't been separately confirmed.
- **If a single `AzureIoT.loop()` call itself blocks for close to 8 seconds** (an unusual MQTT/network stall, not a hang), the watchdog could fire mid-operation once enabled. Accepted tradeoff given megaAVR-0's ~8s period is close to its own hardware ceiling, not a bug -- part of why `enableWatchdog()` is opt-in rather than automatic.
- **`extractJsonString()`** (in `dps_client.cpp`) is a lightweight substring search, not a full JSON parser -- matches Azure's documented DPS response format but isn't robust to Azure changing that structure. The writable-property feature's `extractJsonBool()`/`extractJsonNumber()` (in `AzureIoT.cpp`) are the same kind of substring search, with the same tradeoff.
- **Auto-threshold examples (18/19) use hysteresis, not a single threshold**, to avoid flicker when a reading naturally hovers near the boundary (real risk with DHT11's documented +/-2 degC / +/-5% accuracy). Copying this pattern with a single threshold instead reintroduces that flicker risk.
- **If `begin()` is ever called more than once** (unusual -- it's normally called once in `setup()`) **with a different `modelId` the second time, the stale one from the first call isn't cleared** -- `s_modelId` is only overwritten when a non-null `modelId` is passed, not reset to empty otherwise. Not a concern for the normal single-`begin()`-call pattern every example uses, but worth knowing if you ever restructure a sketch to call `begin()` conditionally.
- **Uno WiFi Rev2's ATECC608 crypto chip goes unused** -- authentication is a software-computed SAS token (symmetric key), not the hardware-backed X.509 approach the chip is designed for. That'd be a bigger redesign. (ESP32 has no equivalent hardware secure element in most variants, so this doesn't apply there.)
- **`onRemoteTelemetry()`/`azureiot_poll_remote_telemetry()` (v1.1) is NOT YET TESTED on real hardware.** Everything about it has been verified at the logic/spec level -- the exact REST endpoint and response shape cross-checked against Microsoft's current official docs, the JSON parsing tested against the exact example values from those docs, the poll-interval clamping and registration cap tested via the real code -- but nobody has yet run a device that maintains its own MQTT connection *and* periodically opens a second, separate HTTPS connection for this feature at the same time. The specific things worth watching for on a first real test: whether polling (even at the 1000ms floor) causes the primary MQTT keepalive to be missed, and general memory pressure from two TLS-capable connections' worth of buffers coexisting. Test this before relying on it, the same way the watchdog needed its own dedicated hardware test before being trusted.

## What's been verified (and how)

- **Crypto core** (`sha256.c`/`b64url.c`/`sas_token.c`): checked against NIST/RFC test vectors and, separately, byte-for-byte against an independent Python (`hashlib`/`hmac`/`base64`) implementation of Azure's documented SAS-token algorithm, across 16 randomized cases. Fully portable C, compiled clean for both AVR and ARM targets.
- **API usage, both platforms**: every WiFiNINA/PubSubClient call and every ESP32 WiFi/WiFiClientSecure/Esp.h call checked against each library's actual current source (arduino-libraries/WiFiNINA and espressif/arduino-esp32, respectively) before being used.
- **Azure root CA certificate (ESP32 only)**: copied verbatim from Microsoft's official Azure IoT C SDK (`Azure/azure-iot-sdk-c`, `certs/certs.c`), not reconstructed by hand. Independently verified with `openssl x509`: both embedded certificates parse as valid, correctly self-signed root CAs, and the DigiCert Global Root G2 certificate's SHA-1 fingerprint matches an independently-sourced reference.
- **`publish()`/`loop()` staging and flush logic**: directly unit-tested -- overwrite semantics, correct clearing after a send, multiple keys combined into one message, and 50 rapid calls collapsing to one field per key (confirming the bounded send-rate design actually holds).
- **`publishText()`'s JSON escaping**: unit-tested against 9 cases (plain text, embedded quotes, backslashes, control characters, empty strings, buffer-boundary behavior) -- all correct, confirmed byte-for-byte, including end-to-end payload shape.
- **Long-run escalation logic, both platforms**: the exponential-backoff arithmetic isolated and checked standalone, then the full Wi-Fi-escalation and provisioning-backoff paths tested end-to-end through the real, unmodified `AzureIoT.cpp` against both a WiFiNINA-signature mock and an ESP32-signature mock -- identical behavior confirmed on both.
- **All twenty example sketches** syntax/type-checked against a mock covering the Arduino core calls and the `AzureIoT` public API they actually use. The original six (now renumbered 1, 4, 7, 8, 9, 10) additionally went through a fuller WiFiNINA/PubSubClient-signature mock and an end-to-end run on both platforms, including a live check that the ESP32 path correctly wires the embedded certificate into `WiFiClientSecure::setCACert()`.
- **The writable-property feature** (`onBoolProperty()`/`reportBoolProperty()`): the exact MQTT topic strings (`$iothub/twin/PATCH/properties/desired/#`, `$iothub/twin/res/#`, `$iothub/twin/GET/?$rid=...`, `$iothub/twin/PATCH/properties/reported/?$rid=...`) cross-checked against Microsoft's official IoT Hub MQTT documentation and two independent third-party MQTT-direct implementations -- all matched exactly. The JSON-parsing logic functionally tested against realistic Azure twin JSON: a live incremental patch, a full twin document with both `desired`/`reported` present, multiple properties in one patch, an absent property, and an empty ack response -- all passing, including that `desired` is found before `reported` even when both contain the same property name.
- **The twin `$version` extraction**: verified against 6+ authoritative real Azure twin JSON examples (Microsoft Q&A, Azure IoT SDK for Java's own docs, Azure Samples). `"$version"` (lowercase, exact) appears exactly once per section, always a sibling of `$metadata`, never nested inside it -- `$metadata` uses the differently-named `$lastUpdatedVersion` for per-property tracking, so there's no collision risk with the first-occurrence search.
- **Hardware watchdog** (`AzureIoT.enableWatchdog()`): register/API usage cross-checked against multiple independent sources per platform. megaAVR-0: confirmed via an AVR Freaks forum thread and megaTinyCore/DxCore's own published reset-handling reference docs that avr-libc's `wdt_enable(WDTO_xxx)` wrapper produces the wrong period on this chip family, and that the direct `_PROTECTED_WRITE(WDT.CTRLA, WDT_PERIOD_8KCLK_gc)` write is correct. ESP32: confirmed arduino-esp32 3.x requires the newer `esp_task_wdt_config_t`-struct API (a real breaking change from 2.x). Register-write pattern also syntax-checked against stubbed real macro/struct shapes.
- **`reportBoolProperty()`'s retry buffer**: functionally tested with a test that includes `AzureIoT.cpp` directly as a single translation unit (no drift risk) -- 6 scenarios under ASan/UBSan: immediate send, staging while disconnected, overwrite semantics, retry on reconnect, safe no-op retry, and the 16-slot cap holding without overflow. Not committed into `test/` -- needs a heavier Arduino/PubSubClient mock than `test_crypto.c`, more ongoing-maintenance surface than this repo's trimmed test suite is meant to carry (see `test/README.md`).
- **The model-id feature, re-checked in a dedicated follow-up audit pass** (not just when first written): the URL-encoded model ID and DPS JSON payload were re-verified byte-for-byte against the documented Azure convention using the real model ID from actual hardware testing. An adversarial test with a worst-case 127-character, all-colon model ID (maximum possible 3x URL-encoding expansion) confirmed under ASan/UBSan that the full username still fits with real margin (513 of 700 bytes used). One real gap found and fixed during this pass: the `azureiot_url_encode()` call had no truncation check, unlike every other buffer operation in the file -- safe today given the buffer sizing, but inconsistent, so a check was added for consistency and future-proofing.
- **Real hardware confirmation, Uno WiFi Rev2**: a real user's board has now confirmed, end-to-end, on actual hardware: Wi-Fi connect, DPS provisioning (including with a model ID declared), MQTT connect, telemetry publish (`temperature`/`touch`/`button` all showing correctly in IoT Central), and the full writable-property round-trip (dashboard toggle → device receives it → callback fires → device acks it → Azure confirms the ack → dashboard shows "Accepted") -- captured verbatim in Serial output, not just inferred. Several real bugs were found and fixed specifically because of this real-hardware testing (see the bug history above) that no amount of x86-mock testing could have caught.
- **Remote telemetry polling (v1.1)**: the exact REST endpoint (`GET /api/devices/{id}/telemetry/{name}`) and response shape were cross-checked against Microsoft's current official documentation (not a cached/older version), including the exact authorization header format (`Authorization: SharedAccessSignature ...`, no `Bearer` prefix -- easy to get subtly wrong, checked directly against the docs rather than assumed) and the built-in role IDs (`Administrator`/`Operator`/`Builder`) confirmed via the real `GET /api/roles` example in the docs. The JSON-value parsing (`extractJsonFloat()`) was tested against the exact example response values from those docs, plus integer, negative, and missing-key edge cases -- all pass under ASan/UBSan. The poll-interval clamping (values below 1000ms get raised to 1000ms) and the 4-watch registration cap were both tested against the real code via single-TU inclusion, not just reasoned about. **Not yet tested on real hardware** -- see Known Limitations above for specifically what to watch for on a first real test.
- **Still not verified on real hardware**: ESP32 (this sandbox has no Xtensa/RISC-V toolchain, and no real-hardware testing has been reported back yet). On Uno WiFi Rev2, every major feature -- connectivity, telemetry, self-healing, writable properties (both directions), and the hardware watchdog -- has been confirmed working on real hardware. The remote-telemetry polling feature (v1.1) is the one exception, added most recently and not yet run on a real board.

## Bug history (found and fixed during audits)

- **Twin-topic buffers undersized for `unsigned long`'s full range**: `ackBoolProperty()`, `reportBoolProperty()`, and `requestFullTwin()`'s MQTT topic buffers were sized for a "typical" request-ID length, not the actual worst case. Reproduced directly: once the shared request-ID counter reached 100, the topic silently truncated mid-number, corrupting `$rid`. Not a memory-safety bug (`snprintf` bounds itself, confirmed clean under ASan/UBSan at the true max, 4294967295), but a real protocol-correctness one. Fixed with proper sizing + truncation checks.
- **`resourceUri` buffer undersized** in `refreshSasTokenIfNeeded()`: sized for a "typical" hub hostname + device ID, not both fields' actual documented maximums at once (200 needed, 192 available). Found via `-Wformat-truncation=2`, not a routine build flag. Fixed the same way.
- **`username` buffer had only 4 bytes of real margin** in `ensureMqttConnected()` against the same worst-case field lengths -- not actually broken today, but fragile and the last `snprintf` without a truncation check. Fixed for consistency/future-proofing.
- **`17_ClapToToggleLed` was level-triggered with a cooldown, not edge-triggered**: sustained loud noise (not an actual clap) would retoggle the LED every cooldown period instead of being treated as one event. Fixed with rising-edge detection, still cooldown-gated for a real clap's decaying tail.
- **`18_AutoNightLight`/`19_ComfortAlarm` had no hysteresis**: real flicker risk at the threshold boundary, especially given DHT11's +/-2 degC accuracy. Fixed with dual (low/high) thresholds.
- **`19_ComfortAlarm` originally overloaded `"buzzerOn"`** as both a writable mute-control and telemetry -- same name, two meanings depending on direction. Caught during design, split into a `"muted"` property + separate `"buzzerOn"` telemetry.
- **Design flaw caught before shipping** (not a bug in committed code): a naive "pet the watchdog once per `loop()`" design would have conflicted with `connectWiFi()`'s multi-minute Wi-Fi-reconnect retry tiers. Fixed in design before merging.
- **`strtof()` doesn't exist in this board's avr-libc** (v1.1.0's `remote_telemetry.cpp`, found via a real compile error on actual Uno WiFi Rev2 hardware -- the first real-hardware test of v1.1.0). `strtof()` is a fairly standard C function on desktop platforms and is described as existing in avr-libc's own general documentation, but the actual bundled `stdlib.h` for this board's toolchain doesn't declare it (confirmed by finding the real avr-libc 2.0.0 header source, which declares `strtod()` and `atof()` but not `strtof()` in the same file) -- some avr-libc versions apparently lack it, or gate it behind a feature-test macro Arduino's build doesn't set. Same general class of gap as the earlier `%f`/`snprintf` avr-libc limitation this library already worked around elsewhere. Fixed by using `strtod()` (parses as `double`, cast down to `float`) instead -- confirmed present in the same real header that was missing `strtof()`. This shipped in 1.1.0 and was fixed in 1.1.1 -- anyone who installed 1.1.0 on a Uno WiFi Rev2 and tried to use `onRemoteTelemetry()` would have hit this exact compile error; ESP32 was unaffected (its C library does provide `strtof()`).
- **Telemetry messages were missing the content-type/content-encoding declaration** on the MQTT topic (`flush()` and `publishText()`). Azure IoT Hub still accepts and forwards an undeclared message (so "last data received" updates in IoT Central), but IoT Central specifically needs `$.ct=application%2Fjson&$.ce=utf-8` declared to know it's allowed to parse and display the payload -- without it, messages can arrive successfully but never show up in Raw Data, with no error anywhere. Found via a real user's hardware testing (device connected, telemetry not appearing), confirmed against Microsoft's own IoT Hub message-format documentation before fixing. The topic buffers (96 bytes) also had to grow (to 160) to fit the added declaration.
- **`onBoolProperty()`/`reportBoolProperty()` never declared the device's model ID** during DPS registration or the IoT Hub MQTT connect -- the first real end-to-end hardware test of the writable-property feature showed the dashboard toggle going to "pending" and staying there, with the physical LED never responding at all. Per Microsoft's own IoT Plug and Play device developer guide: "Update your device or module to announce the model-id as part of the device connection," and "IoT Central devices that connect through DPS should follow IoT Plug and Play conventions and send their model ID when they register -- IoT Central uses the model ID to assign the device to the correct device template." Plain telemetry doesn't need this (an enrollment group can have a fixed template configured administratively, which is how the device ended up correctly template-assigned despite never sending a model ID) -- but writable properties are Plug-and-Play-specific and, on real hardware, simply didn't arrive at all without it declared. Fixed by threading an optional `modelId` parameter through `begin()` to both the DPS registration payload (`{"payload":{"modelId":"..."}}`) and the IoT Hub MQTT connect username (`&model-id=<url-encoded>`), verified byte-for-byte against the documented convention with isolated string tests before shipping.
  - **CONFIRMED WORKING on real hardware** after a second real-hardware test round: with the model ID correctly set AND after adding debug logging (see below) to see the actual message exchange, a full round-trip was observed and captured verbatim -- device requests the twin, receives `{"desired":{"$version":6,"ledState":true},...}`, invokes the sketch's callback, sends an ack, and Azure replies `$iothub/twin/res/204/?$rid=2&$version=8` (HTTP 204, accepted). The IoT Central dashboard changed from "Pending" to "✓ Accepted" and the physical LED responded correctly. The first hardware test (model ID alone, no logging yet) still showed "stuck pending" -- the debug logging added afterward was what let this be confirmed as actually fixed, rather than continuing to guess.
  - **CORRECTION, found immediately after fixing the real off-by-one bug below**: once the actual root cause (the `strncmp` length bug) was fixed, a clean follow-up test with the model ID REMOVED again showed writable properties working perfectly -- both the initial twin fetch and multiple live toggle flips, with zero model ID declared. This means the model ID was very likely never actually necessary for this specific setup; the off-by-one bug fixed below is the complete, sufficient explanation for the original "stuck pending" symptom. The model-id feature remains in the library (real, Microsoft-documented, harmless to include, and may matter for different enrollment-group configurations that rely on model-id-based automatic template assignment rather than a fixed template), but the documentation and code comments were corrected to stop overstating it as required for writable properties to function -- it isn't, at least not universally. Lesson: a plausible, well-evidenced hypothesis backed by official documentation still needs a controlled test (with the *other* variable removed) before being confirmed as the actual cause of an observed symptom, not just "shipped and the retest happened to pass."
- **A second, more serious real bug, found immediately after the one above thanks to the new debug logging**: `mqttMessageCallback()`'s `isDesiredPatch` check used `strncmp(topic, "$iothub/twin/PATCH/properties/desired", 38)` -- but that string literal is exactly **37** characters, not 38. Comparing 38 bytes meant the comparison included the literal's own NUL terminator (byte 37) against whatever real character sits there in an actual incoming topic -- always `/` (every real topic continues `.../desired/?$version=N`), never `\0`. Since `'\0' != '/'`, this check **could never succeed for any real live PATCH notification, ever** -- a 100%-reproducible, complete failure of the "cloud pushes a change to an already-connected device" path. This is exactly why the very first working demonstration only worked right after a fresh connect (the twin GET path uses a separate, correctly-sized 17-byte check for `"$iothub/twin/res/"`, which is unaffected) but a toggle flipped while the device was already connected and idle never did anything until the next reconnect. Reported directly by a real user noticing "resetting the board picks up pending changes, but the LED never reacts without a reset." Fixed: `38` -> `37`.
  - **Root cause of why this slipped through multiple prior audit passes**: the earlier "property-parsing functional test" (`test_property_parsing.cpp`) only tested `extractJsonBool()`/`extractJsonNumber()` directly against pre-selected string content -- it never actually exercised `mqttMessageCallback()`'s topic-prefix matching logic with a realistic topic string. A bug in message *routing* (does this topic even get processed at all) is invisible to a test that only checks message *parsing* (given the right content, is it parsed correctly) -- these are different layers, and only one had a test. Fixed going forward: a new test (`test_topic_matching.cpp`, in the same single-TU-inclusion style as the retry-buffer test) calls the real `mqttMessageCallback()` with the exact topic string captured from real hardware (`"$iothub/twin/PATCH/properties/desired/?$version=7"`), and confirms the registered callback actually fires. Verified this test would have caught the original bug: reverting the fix and re-running it reproduces the exact failure (`assert(g_callbackFired == true)` fails). Not committed into `test/` for the same reasons as the retry-buffer test (needs the heavier mock) -- but worth noting explicitly since it's the test that closes the actual coverage gap that let this bug through.
- **Zero debug visibility anywhere in the writable-property pipeline** was itself a real gap, found only because the model-id fix's first real-hardware test left no way to tell *why* the toggle was still stuck. Added logging for: `subscribe()` return values (never checked before -- a silent failure here would explain the exact "stuck pending forever, no errors" symptom), `requestFullTwin()`'s publish result, every message `mqttMessageCallback()` receives (topic + body) and what it decided to do with it, and `ackBoolProperty()`'s publish result. This is what turned the second hardware test from "still doesn't work, no idea why" into a captured, verifiable, successful exchange.
